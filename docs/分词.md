## PostgreSQL 中文分词插件对比

目前 PG 生态主要有以下 4 款中文分词方案：

### 1. zhparser（⭐ 858）—— 最主流

| 项目 | 说明 |
|---|---|
| 底层引擎 | SCWS（简易中文分词系统） |
| 分词原理 | **纯词典匹配**，基于词频统计 + 规则 |
| 新词支持 | ❌ 不自动识别新词，需手动添加自定义词典 |
| 自定义词典 | `zhparser.extra_dicts = 'dict.txt'`，支持 TXT/XDB 格式 |
| 亮点 | 成熟稳定，社区最大，PG9.2+ 全兼容 |
| 劣势 | 词典外的词无法正确切分，"直播带货"可能被切成"直播 / 带货"也可能切成"直 / 播带 / 货" |

自定义词典 + 新词示例：

```
# /path/to/share/tsearch_data/dict_extra.txt
直播带货 5.0 3.0 n
大模型 5.0 3.0 n
AIGC 5.0 3.0 eng
```

```sql
-- 支持运行时动态增删词（2.1+版本，数据库级别）
INSERT INTO zhparser.zhprs_custom_word VALUES ('大模型');
SELECT sync_zhprs_custom_word();  -- 同步后新连接生效
```

---

### 2. pg_jieba（⭐ 412）—— 新词识别最强

| 项目 | 说明 |
|---|---|
| 底层引擎 | jieba（结巴分词） / CppJieba |
| 分词原理 | 词典 + **HMM（隐马尔可夫模型）统计学习** |
| 新词支持 | ✅ **自动新词发现**，HMM 基于字间转移概率推断新词 |
| 自定义词典 | `jieba_user.dict`，权重 + 词性格式 |
| 亮点 | 4 种分词模式灵活切换，新词处理最突出 |
| 劣势 | 依赖 C++11，PG16+ 兼容性需关注 |

**4 种分词模式：**

| 模式 | 配置名 | 说明 | 新词能力 |
|---|---|---|---|
| MP（精确） | `jiebamp` | 纯词典，最精确的切分 | ❌ |
| HMM | `jiebahmm` | 纯统计，基于字频发现新词 | ✅✅ 最强 |
| **Mix（混合）** | **`jiebacfg`** | 词典 + HMM，**推荐** | ✅ |
| Query（搜索） | `jiebaqry` | Mix + 细粒度再切，提高召回 | ✅ |

**实际效果对比（"大模型时代来了"）：**

```
-- zhparser（词典中没有"大模型",默认分词）
'大' & '模型' & '时代' & '来'     -- 切碎了

-- pg_jieba HMM 模式
'大模型' & '时代' & '来'          -- 自动识别新词！
```

**配置示例（HMM 新词发现）：**

```sql
CREATE EXTENSION pg_jieba;

-- 推荐用 jiebaqry（搜索场景）或 jiebacfg（通用场景）
CREATE TEXT SEARCH CONFIGURATION chinese_jieba (PARSER = pg_jieba);
ALTER TEXT SEARCH CONFIGURATION chinese_jieba
    ADD MAPPING FOR n,v,a,eng WITH simple;

-- 配合 pg_textsearch 使用
CREATE INDEX docs_idx ON docs USING bm25(content)
    WITH (text_config='chinese_jieba');
```

**自定义词典格式** (`tsearch_data/jieba_user.dict`)：

```
大模型 10 n
AIGC 5 eng
直播带货 8 n
```

### 3. pg_bigm（⭐ 146）—— Bigram 无词典方案

| 项目 | 说明 |
|---|---|
| 底层引擎 | 基于 **2-gram（二元组）** 的字符级索引 |
| 分词原理 | 不对中文做"词级"切分，而是将所有相邻 2 字的组合作为索引单元 |
| 新词支持 | ✅ 天然支持，任何新词都会被其组成 bigram 覆盖 |
| 亮点 | 不需要词典，语言无关，安装即用 |
| 劣势 | 精度低，伪阳性多，"数据库"会匹配到"数据库"也匹配到"数据/据库/库房" |

**原理示例：**

```
输入："大模型"
Bigram: '大模', '模型'

搜索"大模型" → 匹配所有同时包含 '大模' 和 '模型' 的文档
问题："大数据算法模型" 也会被匹配到
```

```sql
CREATE EXTENSION pg_bigm;
-- pg_bigm 提供 GIN 索引，不直接作为 text_config 解析器
-- 与 pg_textsearch 的 BM25 配合需要额外适配层
```

---

### 4. PGroonga —— 全能搜索引擎

| 项目 | 说明 |
|---|---|
| 底层引擎 | Groonga（C 语言全文搜索引擎） |
| 中文支持 | 内置多种 Tokenizer：`TokenNgram`、`TokenBigram`、`TokenMecab` |
| 新词支持 | NGram/Bigram 模式天然支持；MeCab 模式依赖词典 |
| 亮点 | 零 ETL，支持所有语言，性能极高，自带索引引擎 |
| 劣势 | 独立索引引擎，**无法作为 pg_textsearch 的 text_config 使用** |

PGroonga 走的是独立索引路径，不与 PostgreSQL 的 `CREATE TEXT SEARCH CONFIGURATION` 体系兼容，因此**无法与 pg_textsearch 的 BM25 索引配合使用**。

## 综合推荐

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| **通用中文搜索** | pg_jieba（`jiebacfg`） | HMM 自动发现新词 + 支持自定义词典，平衡精度和召回 |
| **高精度场景 + 可控词典** | zhparser | 纯词典，分词稳定可控，社区成熟 |
| **多语言混合 / 零配置** | pg_bigm | 不需要词典，日语、韩语也能用 |
| **需要独立搜索引擎** | PGroonga | 性能最强，但不是 PG 标准分词器 |

**对新词支持最强的组合：**

```
pg_jieba (jiebacfg/jiebaqry) + 自定义词典 → 自动发现新词 ↗ / 手动补充特定领域词
```

HMM 基于字间转移概率进行新词发现，对于网络新词（"绝绝子""疯狂星期四"）和领域术语（"向量数据库"）有明显效果，是当前 PG 生态中对新词最友好的方案。
