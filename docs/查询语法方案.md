## 核心矛盾：WHERE 要求布尔值，BM25 是连续值

PostgreSQL 的 WHERE 子句 **只接受布尔表达式**：


BM25 的天然语义是"打分排序"而非"是/否匹配"。所以要把排序能力伪装成过滤语法，就需要魔术。

### 路线三：双操作符方案（pgvector 风格，折中）

```sql
-- @@ 做布尔过滤：文档包含至少一个查询词 → true
-- <-> 做排序：返回 BM25 距离（越小越相关）
SELECT *, a.b <-> '关键词' AS score
FROM A
WHERE a.b @@ '关键词'
ORDER BY score
LIMIT 10;
```

| 操作符 | 类型签名 | 作用 | 在 Operator Class 中的角色 |
|---|---|---|---|
| `@@` | `text @@ query_t → boolean` | 过滤：文档含任一查询词 | Strategy 0（Search） |
| `<->` | `text <-> query_t → float8` | 排序：BM25 距离 | Strategy 1（ORDER BY） |

**优点：**
- `@@` 是真正的布尔操作符，不需要 parser hook 改写
- 两个操作符都注册在同一个 Operator Class 中，共享一次索引扫描
- 语义清晰：`@@` 问"有没有"，`<->` 问"多相关"

**`amgettuple` 需要同时处理过滤和排序：**

```c
bool amgettuple(IndexScanDesc scan, ScanDirection dir) {
    // 首次调用：执行 BM25 评分
    if (first_call) {
        execute_bm25_scoring(scan);  // → result_ctids[], result_scores[]
    }

    // 遍历结果，跳过不满足 @@ 过滤的文档
    while (has_more) {
        ctid = result_ctids[pos];
        
        // 检查 @@ 过滤（Strategy 0）
        if (has_scan_key_for_@@) {
            if (!document_has_any_query_term(ctid, query)) {
                pos++; continue;
            }
        }

        // 填入 ORDER BY 分数（Strategy 1）
        scan->xs_orderbyvals[0] = -result_scores[pos];
        pos++;
        return ctid;
    }
}
```

**这个方案不需要 parser hook**，但需要：
- `@@` 操作符的过滤逻辑（可以是简单的"文档是否含任一查询词"）
- `amgettuple` 中同时处理 Search 和 ORDER BY 两种 scan key
- `amrescan` 中区分两种 key

### 路线二：%= + Hook → 规划略重，执行相同

```
用户写 WHERE col %= '关键词' ORDER BY score LIMIT 10
     ↓
post_parse_analyze_hook 遍历 Query 树，识别 %=
     ↓
删掉 %=，构造 ORDER BY col <@> bm25query(...)
     ↓
planner_hook 后处理计划树，替换评分表达式
     ↓
执行 = 和路线一完全相同
```

| 阶段 | 开销 | 备注 |
|---|---|---|
| 规划 | 多个 hook 遍历 Query 树 + 计划树 | **微量增加**（微秒级），生产环境可忽略 |
| 执行 | 与路线一完全相同 | ✅ |

### 路线三：双操作符 → 有潜在陷阱

```sql
SELECT *, col <-> '关键词' AS score
FROM docs
WHERE col @@ '关键词'
ORDER BY score LIMIT 10;
```

| 阶段 | 开销 | 风险 |
|---|---|---|
| 规划 | 标准路径，无 hook | ✅ |
| 执行 | 与路线一相同 | ⚠️ 但规划器可能不选 BM25 索引 |

**陷阱在于规划器的自由度：**

```
规划器看到两个操作符 → 有两种选择：

方案 A（最优）:
  → IndexScan on bm25_idx
    用 @@ 过滤 + <-> 排序，一次扫描搞定

方案 B（次优）:
  → Bitmap Index Scan on gin_idx    （用 @@）
  → Sort by <->                       （再排序）
  BM25 索引根本没用上！
```

一旦规划器选了方案 B，性能可能差一个数量级。而路线一/二因为只有一个 ORDER BY 操作符，规划器**没有选择的余地**——只能走 BM25 索引扫描。

## 方案一：自定义操作符（推荐，不用 `=`）

### 方案 1a：专用操作符 `@@` + `simple` 配置

```sql
-- 用户语法
SELECT * FROM A
WHERE a.b @@ '关键词'
ORDER BY score
LIMIT 10;
```

内部实现：`@@` 不是假的（不像 `%=` 永远返回 true），而是一个**真正的布尔操作符**。

```c
// @@ 操作符实现：文档是否包含任一查询词（精确匹配）
Datum text_search_exact_match(PG_FUNCTION_ARGS) {
    text   *doc   = PG_GETARG_TEXT_PP(0);
    text   *query = PG_GETARG_TEXT_PP(1);
    
    // 用 simple 配置分词（无词干提取，保持原形）
    tsvector = to_tsvector("simple", doc);
    tsquery  = plainto_tsquery("simple", query);
    
    // 如果 tsquery @@ tsvector 匹配 → true
    PG_RETURN_BOOL(tsvector_matches(tsvector, tsquery));
}
```

| 组件 | 说明 |
|---|---|
| `@@` | `text @@ text → boolean`，注册在 Operator Class 中 Strategy 0 |
| `<@>` 或 `<->` | `text <@> text → float8`，Strategy 1（ORDER BY） |

```sql
-- Operator Class 注册
CREATE OPERATOR CLASS text_bm25_exact_ops DEFAULT FOR TYPE text USING bm25 AS
    OPERATOR  1  <@> (text, bm25query) FOR ORDER BY float_ops,   -- 打分排序
    OPERATOR  2  @@  (text, text) FOR SEARCH,                     -- 精确匹配过滤
    FUNCTION  8  bm25_score(text, bm25query);
```

**amgettuple 同时处理过滤和排序：**

```c
bool amgettuple(IndexScanDesc scan, ScanDirection dir) {
    if (first_call) {
        execute_bm25_scoring(scan);  // BM25 评分 → sorted_result_ctids[]
    }

    while (has_more) {
        ctid       = sorted_result_ctids[pos];
        raw_score  = sorted_result_scores[pos];

        // @@ 过滤（精确匹配）：文档必须包含至少一个查询词的精确形式
        if (has_@@_filter && !doc_has_exact_term(ctid, query_terms)) {
            pos++; continue;
        }

        scan->xs_orderbyvals[0] = -raw_score;  // BM25 分数
        pos++;
        return ctid;
    }
}
```

**不需要 parser hook**，纯靠 PostgreSQL 原生的 Operator Class 机制驱动。

## 方案二：Planner Hook 劫持 `=`（危险，不推荐）

如果想强行用 `=`，可以模仿 `%=` 的做法：

```c
// post_parse_analyze_hook 中
static void rewrite_exact_eq_query(Query *query) {
    // 遍历 WHERE，找这个模式：
    //   WHERE text_column = 'literal'
    //   AND query->sortClause 包含 "score"
    
    // 确认列上有 BM25 索引后：
    //   1. 删除 WHERE col = 'literal'
    //   2. 替换为 ORDER BY col <@> bm25query('literal', index_oid)
    //   3. 注入 bm25_get_current_score() AS score

```

**为什么不推荐：**

| 风险 | 说明 |
|---|---|
| `=` 无处不在 | `WHERE a.b = c.d`（JOIN）、`WHERE a.b = $1`（参数化查询）、`WHERE a.b = (SELECT ...)` 都会被误伤 |
| 区分难度大 | 很难在 hook 中区分"精确搜索的 `=`" 和"普通等值比较的 `=`" |
| 调试地狱 | 用户写 `WHERE col = 'something'` 发现有奇怪副作用，极难定位 |
| 与其他扩展冲突 | 任何处理 `=` 的其他扩展都可能和你打架 |

**唯一安全的判断条件：同时满足**

```
① col = 'literal_string'         — 右操作数是字符串字面量（Const 节点）
② ORDER BY score                  — 排序目标名必须是 "score"
③ col 上有 BM25 索引               — Planner hook 中二次确认
```

## 方案三：自定义类型绕过（优雅但改 DDL）

创建一个 `bm25text` 类型，它和 `text` 有隐式转换但不能互操作 `=`：

```sql
CREATE TYPE bm25text (
    INPUT   = bm25text_in,
    OUTPUT  = bm25text_out,
    INTERNALLENGTH = VARIABLE
);

-- bm25text = text → 精确匹配 + BM25 评分
CREATE OPERATOR = (
    LEFTARG  = bm25text,
    RIGHTARG = text,
    PROCEDURE = bm25_exact_match_scoring   -- 同时做匹配 + 评分
);

-- bm25text = bm25text → 委托给内置 texteq
CREATE OPERATOR = (
    LEFTARG  = bm25text,
    RIGHTARG = bm25text,
    PROCEDURE = texteq   -- 走 PostgreSQL 内置比较
);
```

**用户用法：**

```sql
-- 建表时列类型用 bm25text
CREATE TABLE A (
    id int,
    b bm25text    -- 而不是 text
);

-- 查询语法就是你想要的
SELECT * FROM A
WHERE a.b = '关键词'
ORDER BY score
LIMIT 10;
```

**工作原理：**

| 查询部分 | 操作符 | 行为 |
|---|---|---|
| `WHERE a.b = '关键词'` | `bm25text = text` | 精确分词匹配，返回 boolean |
| `ORDER BY score` | 桩函数 `get_score()` | 读 BM25 缓存分数排序 |

**优点：**
- 语法的确是你写的 `WHERE a.b = '关键词'`
- `=` 不会误伤普通 `text` 列
- 不需要 parser hook

**缺点：**
- 必须修改表结构（`ALTER TABLE ... ALTER COLUMN ... TYPE bm25text`）
- 和 `text` 类型的生态（函数、操作符）不能无缝互操作
