The user wants to implement phrase search (短语检索) with syntax like `WHERE col == '关键词'`. 

The key challenge with phrase search vs regular word search:
1. **Word search** (like `@@` or `%=`): "hello world" matches any document containing "hello" OR "world"
2. **Phrase search**: "hello world" matches only documents where "hello" comes immediately before "world"

The critical difference in implementation: phrase search requires **position information** (term positions within documents), while regular BM25 only stores **term frequencies** (how many times each term appears).

pg_textsearch doesn't support phrase queries because it stores only term frequencies, not term positions. Let me verify this from the README.

From the README:
> The BM25 index stores term frequencies but not term positions, so it cannot natively evaluate phrase queries

So implementing phrase search requires storing position data in the index.

Let me think about the implementation approaches:

1. **Positional BM25 index**: Store positions alongside term frequencies in posting lists
2. **Post-filter approach**: Use BM25 for ranking, then post-filter with `ILIKE '%phrase%'` 
3. **Hybrid approach**: BM25 index for ranking + GIN tsvector index for position-based screening

The `==` operator syntax is good for phrase matching (double equals suggests "exactly this phrase").

Let me think about the implementation details:

### Index Structure Change
Each posting entry needs to store not just doc_id and term_freq, but also positions:
```
Term "hello" → posting list: [(doc1, tf=2, pos=[5, 23]), (doc3, tf=1, pos=[0]), ...]
Term "world" → posting list: [(doc1, tf=1, pos=[6]), (doc2, tf=1, pos=[15]), ...]
```

For phrase query "hello world":
1. Merge posting lists for "hello" and "world"
2. For each shared doc_id (doc1), check if "world" comes immediately after "hello" (pos=6 follows pos=5)
3. Score only if positions are adjacent (window=1)

### Scan Execution
The amgettuple flow:
1. Query text → tokenize into terms
2. For each term, read posting lists including positions
3. BM25 scoring + phrase position check
4. Return in score order

### Implementation Options

**Option A: Store positions in the BM25 index**
- Modify posting storage to include position arrays
- Increased index size (positions take significant space)
- Can do exact phrase matching inline with scoring

**Option B: Post-filter approach (pg_textsearch README suggests)**
- BM25 scores normally (no positions needed)
- Post-filter with ILIKE or regex
- Index size unchanged
- Less accurate (false positives before filter)

**Option C: Hybrid with GIN (tsvector)
- BM25 index for ranking
- GIN tsvector index for phrase filtering (tsvector stores positions)
- Use both indexes in the query

Let me provide a comprehensive answer focused on Option A (storing positions in the BM25 index) since that's the most integrated approach for a from-scratch implementation.

The `==` operator is a great choice for phrase search since it visually suggests "exact match."

## 核心差异：词级 vs 短语级

```
词级检索（BM25, 现有）：
  "hello world" → 找含 "hello" 的文档 + 找含 "world" 的文档 → 合并打分
  不需要知道 "hello" 和 "world" 的位置关系

短语检索（需要实现）：
  "hello world" → 找 "hello" 和 "world" 相邻出现的文档
  必须知道每个词在文档中的精确位置
```

pg_textsearch README 明确写了：「BM25 index stores term frequencies but **not term positions**, so it cannot natively evaluate phrase queries」。所以实现短语检索的关键是：**在 posting 中存储位置信息**。

## 索引结构调整

### 改造前（只有词频）

```
Term "hello" → posting list:
  doc_1: tf=2
  doc_3: tf=1
  doc_5: tf=3

Term "world" → posting list:
  doc_1: tf=1
  doc_2: tf=1
```

### 改造后（加位置）

```
Term "hello" → posting list:
  doc_1: tf=2, pos=[5, 23]         ← 第 5 个和第 23 个词是 "hello"
  doc_3: tf=1, pos=[0]
  doc_5: tf=3, pos=[1, 7, 15]

Term "world" → posting list:
  doc_1: tf=1, pos=[6]              ← 第 6 个词是 "world"
  doc_2: tf=1, pos=[42]
```

### Posting 结构变更

```c
// 改造前：只有文档和词频
typedef struct Posting {
    uint32 doc_id;
    uint32 term_freq;
} Posting;

// 改造后：加上位置数组
typedef struct PhrasePosting {
    uint32  doc_id;
    uint32  term_freq;
    uint32 *positions;      // ← 新增：按升序排列的位置列表
    uint16  pos_count;      // ← 位置个数（== term_freq）
} PhrasePosting;
```

**空间代价：** 位置数据约占 term frequencies 的 **3-5 倍**存储空间。

## SQL 定义

```sql
-- == 操作符：短语精确匹配（两个词相邻出现）
CREATE FUNCTION text_phrase_match(text, text) RETURNS boolean
    AS 'MODULE_PATHNAME', 'text_phrase_match'
    LANGUAGE C IMMUTABLE STRICT PARALLEL SAFE;

CREATE OPERATOR == (
    LEFTARG  = text,
    RIGHTARG = text,
    PROCEDURE = text_phrase_match
);

-- <-> BM25 评分操作符（不变）
CREATE OPERATOR <-> (
    LEFTARG  = text,
    RIGHTARG = text,
    PROCEDURE = text_bm25_score     -- BM25 仍然只用词频，不需要位置
);

-- Operator Class：词级 + 短语级共存
CREATE OPERATOR CLASS text_bm25_ops DEFAULT FOR TYPE text USING bm25 AS
    OPERATOR  1  <-> (text, text) FOR ORDER BY float_ops,   -- BM25 排序
    OPERATOR  2  %=  (text, text) FOR SEARCH,                -- 词级模糊
    OPERATOR  3  ==  (text, text) FOR SEARCH;                -- 短语精确
```

## amgettuple 短语实现

```c
bool amgettuple(IndexScanDesc scan, ScanDirection dir) {

    // ====== 阶段 0：解析查询 ======
    if (first_call) {
        char *query_text = extract_query_text(scan);

        // 分词 + 去重，得到查询词列表
        tokenize_query(query_text, &query_terms, &num_terms);
        // 例："hello world" → ["hello", "world"]

        // ====== 阶段 1：BM25 评分（不需要位置，沿用现有逻辑） ======
        for each doc in inverted_index:
            double bm25 = compute_bm25(doc, query_terms);
            candidate_heap_insert(doc, bm25);

        // → result_ctids[]  按 BM25 分数降序排列
    }

    // ====== 阶段 2：遍历候选 + 短语过滤 ======
    while (pos < result_count) {
        ctid  = result_ctids[pos];
        score = result_scores[pos];

        // ─── == 短语过滤（FOR SEARCH, strategy=3） ───
        if (has_phrase_filter) {
            if (!check_phrase_match_for_ctid(ctid, query_terms, num_terms)) {
                pos++; continue;
            }
        }

        // ─── 通过 → 返回 ───
        scan->xs_orderbyvals[0]  = Float4GetDatum(-score);
        scan->xs_orderbynulls[0] = false;
        pos++;
        return true;
    }
}
```

### 核心算法：短语相邻检查

```c
/**
 * 检查一个文档中，查询词是否以相邻顺序出现
 * 例：query = ["hello", "world"] → 检查是否存在 pos("world") == pos("hello") + 1
 *
 * 算法：取最短的 posting list，遍历其每个位置，检查其他词在相邻位置是否也存在
 */
static bool check_phrase_match_for_ctid(
        uint32 ctid, char **query_terms, int num_terms)
{
    if (num_terms <= 1) return true;  // 单词语法上等同于词级

    // 1. 拿到每个词的 posting list 中该文档的位置数组
    uint32 *pos_arrays[MAX_TERMS];
    int      pos_counts[MAX_TERMS];

    for (int i = 0; i < num_terms; i++) {
        PhrasePosting *p = get_posting(query_terms[i], ctid);
        if (!p) return false;         // 有词不在文档中 → 直接失败
        pos_arrays[i]  = p->positions;
        pos_counts[i]  = p->pos_count;
    }

    // 2. 选最短位置的 term 作为锚点
    int anchor = 0;
    for (int i = 1; i < num_terms; i++) {
        if (pos_counts[i] < pos_counts[anchor])
            anchor = i;
    }

    // 3. 遍历锚点的每个位置 → 检查是否形成连续序列
    for (int p = 0; p < pos_counts[anchor]; p++) {
        uint32 candidate_start = pos_arrays[anchor][p] - anchor;
        // 例：anchor=1(hello, pos=5) → 期望 world 在 pos=6
        //          anchor=0(world, pos=6) → 期望 hello 在 pos=5

        bool all_match = true;
        for (int i = 0; i < num_terms; i++) {
            uint32 expected_pos = candidate_start + i;
            if (!position_exists(pos_arrays[i], pos_counts[i], expected_pos)) {
                all_match = false;
                break;
            }
        }

        if (all_match) return true;   // 找到连续序列
    }

    return false;
}

// 二分查找（位置数组已升序）
static bool position_exists(uint32 *arr, int count, uint32 target) {
    int lo = 0, hi = count - 1;
    while (lo <= hi) {
        int mid = (lo + hi) / 2;
        if (arr[mid] == target) return true;
        if (arr[mid] < target) lo = mid + 1;
        else                   hi = mid - 1;
    }
    return false;
}
```

**复杂度：** `O(min_pos_array_len × num_terms × log(pos_count))`，对于常见搜索词（高频词位置多，低频词位置少），取最短 posting 做锚点可以大幅剪枝。

## 完整查询语法体系

实现后，你的插件支持三层检索精度：

```sql
-- 第一层：词级模糊检索
SELECT *, a.b <-> '数据库 性能' AS score
FROM A
WHERE a.b %= '数据库 性能'
ORDER BY score
LIMIT 10;
  → 匹配含 "数据库" OR "性能" 的文档，BM25 打分

-- 第二层：短语精确检索
SELECT *, a.b <-> '数据库 性能' AS score
FROM A
WHERE a.b == '数据库 性能'
ORDER BY score
LIMIT 10;
  → 匹配含 "数据库 性能" 两个词相邻出现的文档，BM25 打分

-- 第三层：组合过滤
SELECT *, a.b <-> '数据库 性能' AS score
FROM A
WHERE a.b %= '数据库 性能'
  AND a.b == '查询优化'
ORDER BY score
LIMIT 10;
  → 文档必须包含 phrased "查询优化"
  → 同时用词级 "数据库 性能" 做 BM25 打分
```

## 对比总结

| | 词级检索 `%=` | 短语检索 `==` |
|---|---|---|
| **索引存储** | posting: doc_id + term_freq | posting: doc_id + term_freq + **positions[]** |
| **索引空间** | 基准 1× | 约 4-6×（位置数组） |
| **扫描复杂度** | O(合并 posting lists) | O(合并 + 位置匹配) |
| **匹配逻辑** | doc 含任一查询词 → 命中 | 所有查询词必须**连续相邻**出现在 doc 中 |
| **"hello world"** | 匹配含 "hello" 或 "world" 的 doc | 只匹配含相邻 "hello world" 的 doc |
| **pg_textsearch 现状** | ✅ 支持 | ❌ 不支持（无位置数据） |

## 一句话建议

```
== 的 phrase 语义需要 positions 数据作支撑。

如果你的索引从零设计 → 建造时就把 positions 存进 posting，
后期加 == 只需查位置相邻即可。

如果索引已经设计好不含 positions → == 只能退化为 post-filter（ILIKE），
精确度和性能都不理想。
```
