## WHERE 过滤 vs ORDER BY 排序：引擎层视角的完整差异

从 PostgreSQL 索引访问方法（Index Access Method）的角度，WHERE 过滤和 ORDER BY 排序是两套**完全独立**的机制。下面从 SQL 定义到执行引擎，逐层拆解。

---

## 第一层：Operator Class 注册差异

```sql
-- pg_textsearch 的 operator class（当前只有 ORDER BY）
CREATE OPERATOR CLASS text_bm25_ops DEFAULT FOR TYPE text USING bm25 AS
    OPERATOR  1  <@> (text, bm25query) FOR ORDER BY float_ops,     -- ← ORDER BY
    FUNCTION  8  (text, bm25query)  bm25_text_bm25query_score;

-- 如果要加上 WHERE 过滤（假设新增 @@ 操作符）
CREATE OPERATOR CLASS text_bm25_ops DEFAULT FOR TYPE text USING bm25 AS
    OPERATOR  1  <@> (text, bm25query) FOR ORDER BY float_ops,     -- 排序
    OPERATOR  2  @@  (text, text)      FOR SEARCH,                  -- 搜索（WHERE）
    FUNCTION  8  (text, bm25query)  bm25_text_bm25query_score;
```

| 维度 | WHERE（`FOR SEARCH`） | ORDER BY（`FOR ORDER BY`） |
|---|---|---|
| 操作符返回值 | **必须** `boolean` | 任意标量类型（`float8`, `int4` 等） |
| strategy number | 任意（PostgreSQL 内核按编号解释语义） | 任意（只需和 AM 内部约定一致） |
| 排序族 | 不涉及 | `float_ops` 等，决定排序比较逻辑 |
| 多个操作符 | 可以有多个 SEARCH 操作符（不同 strategy） | 一个索引仅一个 ORDER BY 操作符 |

**关键约束：** `WHERE` 子句要求布尔表达式，所以 `FOR SEARCH` 的操作符返回值必须是 `boolean`。这就是为什么 `WHERE col <@> 'query'`（返回 `float8`）不能被规划器直接当作过滤条件使用。

## 第二层：Scan Key 传递差异

```c
// IndexScanDesc 结构体包含两套独立的 ScanKey
typedef struct IndexScanDescData {
    int     numberOfKeys;         // WHERE 过滤条件数量
    ScanKey keyData;              // WHERE ScanKey 数组

    int     numberOfOrderBys;     // ORDER BY 条件数量
    ScanKey orderByData;          // ORDER BY ScanKey 数组
    Datum  *xs_orderbyvals;       // 排序值输出（amgettuple 填入）
    bool   *xs_orderbynulls;      // 排序值 NULL 标记
} IndexScanDescData;
```

**ambeginscan 签名：**

```c
IndexScanDesc ambeginscan(Relation index, int nkeys, int norderbys);
//                                              ↑              ↑
//                                          WHERE 数量      ORDER BY 数量
```

**amrescan 签名（重启扫描）：**

```c
void amrescan(IndexScanDesc scan, ScanKey keys, int nkeys,
              ScanKey orderbys, int norderbys);
//               ↑                      ↑
//           WHERE 的 ScanKey        ORDER BY 的 ScanKey
```

两套 ScanKey 在 PostgreSQL 内核中从规划、生成到执行都走**完全独立的通路**：

```
WHERE col @@ '关键词'              ORDER BY col <@> to_bm25query('关键词')
       ↓                                    ↓
   以 OpExpr 出现在 qual 中              以 OpExpr 出现在 indexorderby 中
       ↓                                    ↓
   生成 ScanKey[0]                      生成 orderByData[0]
       ↓                                    ↓
   sk_strategy = 2 (SEARCH)             sk_strategy = 1 (ORDER BY)
   sk_argument = '关键词'::text          sk_argument = '关键词'::bm25query
```

## 第三层：amgettuple 执行差异

这是最关键的差异——**索引扫描函数如何同时处理过滤和排序**：

```c
bool amgettuple(IndexScanDesc scan, ScanDirection dir) {
    // ====== 阶段一：懒加载评分 ======
    if (first_call) {
        // 从 ORDER BY ScanKey 中提取查询文本和索引 OID
        bm25query = extract_bm25query(scan->orderByData[0]);

        // 执行 BM25 评分 → 产生按分数排序的候选列表
        execute_bm25_scoring(bm25query);
        // → result_ctids[]   = [doc3, doc1, doc7, doc2, ...] 按分排序
        // → result_scores[]  = [-8.2, -5.1, -2.3, -1.0, ...]
    }

    // ====== 阶段二：按序遍历候选 ======
    while (pos < result_count) {
        ctid  = result_ctids[pos];
        score = result_scores[pos];

        // ─── WHERE 过滤（仅当有 SEARCH ScanKey 时） ───
        if (has_where_filter) {
            // scan->keyData[0] → `@@` 操作符的 ScanKey
            // 检查：文档 ctid 是否包含查询词？
            if (!doc_matches_terms(ctid, extract_terms(scan->keyData[0]))) {
                pos++; continue;   // ← 不满足过滤，跳过
            }
        }

        // ─── ORDER BY 分数输出 ───
        scan->xs_orderbyvals[0]  = Float4GetDatum(-score);  // 取反适配 ASC
        scan->xs_orderbynulls[0] = false;
        pos++;
        return true;   // ← 返回给 executor
    }
    return false; // 没有更多结果
}
```

**核心差异：**

| | WHERE 过滤 | ORDER BY 排序 |
|---|---|---|
| 输入来源 | `scan->keyData[]` | `scan->orderByData[]` |
| 处理方式 | **跳过低分文档中不满足过滤的** | **填入分数后返回** |
| 输出方式 | 无输出——仅影响哪些元组被返回 | **填入 `xs_orderbyvals[]`**，PG 内核据此维护排序 |
| 对候选顺序的影响 | 可能跳过前 N 个候选 | 决定了候选的顺序本身 |

## 第四层：成本估算差异

```c
void tp_costestimate(...) {
    // 规则一：没有 ORDER BY → 无限成本，禁止使用索引
    if (!path->indexorderbys || list_length(path->indexorderbys) == 0) {
        *indexStartupCost = DBL_MAX;
        *indexTotalCost   = DBL_MAX;
        return;
    }

    // 规则二：读取 metapage 中的实际文档数
    num_tuples = metap->total_docs;

    // 规则三：有 WHERE 过滤时，可以降低行数估算
    if (path->indexquals) {
        // WHERE @@ '关键词' 过滤 → 降低 indexSelectivity
        *indexSelectivity = estimate_filter_selectivity(path->indexquals);
    }

    // 规则四：有 LIMIT 时，启动成本可以大幅降低（BMW top-k）
    if (has_limit) {
        *indexStartupCost *= 0.1;
    }
}
```

**规划器做决定时的差异：**

```
场景 A: ORDER BY col <@> 'q' LIMIT 10           → 只有 ORDER BY
  规划器: IndexScan on bm25_idx（唯一选择）
  
场景 B: WHERE col @@ 'q' ORDER BY col <@> 'q' LIMIT 10   → WHERE + ORDER BY
  规划器: 选项 1 → IndexScan on bm25_idx（@@ 在 bm25 操作符类中）
         选项 2 → BitmapScan on gin_idx(@@) + Sort(<@>)  （如果还有 GIN 索引）
         规划器比较两者的成本——ORDER BY 降低 indexTotalCost 是关键
```

## 完整对比汇总

| 维度 | WHERE 过滤 | ORDER BY 排序 |
|---|---|---|
| **SQL 语义** | 二元判断：行留下还是丢弃 | 连续值：行排第几 |
| **Operator Class** | `OPERATOR N xxx FOR SEARCH` | `OPERATOR N xxx FOR ORDER BY type_ops` |
| **返回类型** | **必须** `boolean` | 任意可排序类型 |
| **ScanKey** | `scan->keyData[0..nkeys-1]` | `scan->orderByData[0..norderbys-1]` |
| **amgettuple 行为** | 真 → 继续返回；假 → 跳过 | 填入 `xs_orderbyvals[]` |
| **输出槽** | 无（隐式影响返回集合） | `xs_orderbyvals[]` + `xs_orderbynulls[]` |
| **成本影响** | 降低 `indexSelectivity`（行数估算） | 降低 `indexStartupCost`（因为有排序路径） |
| **规划器角色** | 可选择其他索引做过滤 | 强制走此索引（唯一排序路径） |
| **ampurgematch** | 涉及（用于 VACUUM 冲突检测） | 不涉及 |

## 工程实践：实现两类操作符共存的套路

```c
// handler.c 中注册
IndexAmRoutine *amhandler() {
    IndexAmRoutine *routine = makeNode(IndexAmRoutine);
    routine->ambuild          = tp_build;
    routine->ambuildempty     = tp_buildempty;
    routine->aminsert         = tp_insert;
    routine->ambeginscan      = tp_beginscan;     // nkeys + norderbys
    routine->amrescan         = tp_rescan;        // keys[] + orderbys[]
    routine->amgettuple       = tp_gettuple;      // 同时处理两者
    routine->amcostestimate   = tp_costestimate;
    routine->ampurgematch     = tp_purgematch;    // 只对 WHERE 操作符
    return routine;
}

// ampurgematch: 告诉 PG 某操作符+strategy 是否匹配
static bool tp_purgematch(Relation index, IndexScanDesc scan,
                          ItemPointer tid, ScanKey key, bool *match) {
    if (key->sk_strategy == 2) {  // SEARCH strategy
        // 检查 tid 是否含查询词 → 填 *match
        *match = doc_has_terms(tid, TermArrayGetTerms(key->sk_argument));
        return true;
    }
    return false;  // ORDER BY strategy → ampurgematch 不处理
}
```

## 总结

```
WHERE 和 ORDER BY 是两条平行线：

  WHERE ≈ "是/否" — 二元门控，recheck 可以纠正，ampurgematch 有语义
  ORDER BY ≈ "多大/多小" — 连续排序，xs_orderbyvals 透传，牺牲 recheck

在 BM25 索引中：
  ORDER BY 是引擎存在的理由（排序必须走索引）
  WHERE 是可选的加速器（可走 B-tree / GIN 等其他索引）
```
